# Enterprise Software Lifecycle Harness Plan

## Problem Frame

Build a repeatable harness for creating enterprise software with AI assistance while preserving planning rigor, implementation quality, offline deployability, and operational maintainability.

The harness itself is the product of this plan. Product-domain behavior is out of scope.

## Scope Boundaries

### In Scope

- Repository-level harness structure.
- AI agent rules.
- Lifecycle documentation.
- Templates and rubrics.
- Plan gate and implementation gate scripts.
- GitHub Actions CI.
- Docker Compose reference deployment path.
- Product starter template.
- Offline bundle generation and verification.

### Out of Scope

- Building a specific business product.
- Kubernetes implementation in v1.
- Private GitHub or paid GitHub features.
- Cloud-managed services as runtime dependencies.
- Product-specific domain models or screens.

### Deferred to Follow-Up Work

- Kubernetes deployment profile.
- Stronger policy-as-code engine.
- Full vulnerability scanner integration.
- Automated GitHub branch protection provisioning.

## Key Decisions

- Use a strong standard path for infrastructure-level decisions.
- Keep product-domain choices out of the harness.
- Use Docker Compose as the v1 deployment reference path.
- Preserve future Kubernetes compatibility by avoiding Compose-only assumptions in templates.
- Use PostgreSQL as the reference database path.
- Allow FastAPI and InfluxDB only through ADR exceptions.
- Use scripts and CI as executable gates, not documents alone.

## Implementation Units

- U1. Repository baseline
  - Files: `README.md`, `AGENTS.md`, `HARNESS.md`, `STRATEGY.md`, `.gitignore`, `.gitattributes`
  - Verification: implementation gate checks required root files.

- U2. Lifecycle documentation
  - Files: `docs/lifecycle/*`
  - Verification: implementation gate checks required lifecycle stage contracts.

- U3. Templates and rubrics
  - Files: `docs/templates/*`, `docs/rubrics/*`, `docs/playbooks/*`
  - Verification: implementation gate checks minimum required templates and rubrics.

- U4. Harness scripts
  - Files: `scripts/*.mjs`
  - Verification: `npm run check` passes.

- U5. Product starter template
  - Files: `templates/product-starter/**`
  - Verification: `npm run compose:validate` passes.

- U6. CI
  - Files: `.github/workflows/ci.yml`
  - Verification: CI contains plan, implementation, Compose, and offline bundle checks.

- U7. Reference monorepo
  - Files: `apps/web`, `apps/api`, `packages/shared`, `packages/config`, `packages/db`, `ops/compose`, `ops/docker`, `ops/runbooks`
  - Verification: root monorepo test and implementation gate pass.

- U8. Toolchain integration and harness coherence
  - Files: `docs/lifecycle/toolchain-integration.md`, `docs/rubrics/harness-coherence.md`, `docs/validation/2026-05-05-round-1-harness-coherence.md`, `scripts/check-toolchain-integration.mjs`
  - Verification: `npm run check:toolchain` and full harness check pass.

## Plan Gate Criteria

- Purpose and non-goals are explicit.
- Infrastructure-level standards are explicit.
- Product-domain choices are not hard-coded.
- Offline/on-prem deployment is addressed.
- Docker Compose is the v1 reference path.
- PostgreSQL default and exceptions are documented.
- Plan and implementation gates are separate.
- CI and scripts are required parts of the harness.

## Implementation Gate Criteria

- Required files and directories exist.
- Required scripts are executable through npm.
- Toolchain integration is documented and checked.
- Root reference monorepo exists and declares workspaces.
- Product starter contains frontend, backend, database, and Compose baseline.
- Compose config validates.
- Offline bundle can be generated.
- Offline bundle checksums can be verified.
- No obvious secrets are committed.

## Risks

| Risk | Mitigation |
|------|------------|
| Harness becomes a product template only | Keep product-domain behavior out of the harness and validate this in docs. |
| Harness becomes documentation-only | Require scripts and CI as executable gates. |
| Offline requirements are discovered too late | Require offline bundle and runtime assumptions in plan and implementation gates. |
| Strong defaults block valid exceptions | Allow exceptions through ADR plus additional verification gates. |
